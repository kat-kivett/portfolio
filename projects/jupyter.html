Coffee Offerings Analysis

A project using Python and Jupyter to clean and explore a dataset created from a sample Starbucks menu. This project demonstrates the need for business knowledge in addition to technical knowledge.

This analysis is meant to showcase the full process from cleansing of data to exploration of various attributes. This would not be a finished product that would be shared with leadership, but would rather be an internal or initial anaysis, informing next steps in analysis and exploration.

Project Overview

Tools Used

Jupyter

Jupyter is a popular choice for analyses like this exploratory dive into the menu information. It provides simple ways to provide context along side the code. The visual interface is simple to use for both technical and business partners.

Libraries

  • pandas
  • numpy
  • matplotlib
  • seaborn
  • re

Pandas does the heavylifting for much data work in Python. The package is very well established and has many, many built in properties that make it ideal for a data analyst. The ability to easily read in a CSV or Excel document as well as other, more integrated database style data sources create an extremely flexible and powerful landscape for fetching and analyzing information.

Matplotlib and Seaborn are the visualization libraries, with Seaborn built on top of Matplotlib. I utilized Seaborn for the aesthetics of the visualizations, however that does come at the cost of the additional customization that is available with Matplotlib. For this analysis there was little need for the fine-grain controls that Seaborn is lacking.

Data Cleansing and Analysis in Python

Kaggle has provided a dataset with nutrition information for a variety of Starbucks drinks. These range in size, in ingredient composition, and in caffeine content. I chose this dataset for a few reasons. One is that while it is generally clean it does require some work to be able to do a quality analysis. The other is that there are nuances to this data that require business knowledge, not just technical knowledge. As part of this EDA we'll walk through cleansing this data, extracting additional information that isn't immediately available, and see how an initial exploratory analysis could inform future, more robust analytics work.

Overall the data is clean and well organized. One observation is that a lot of information is looped into the 'Beverage_prep' category, including cup size, milk type, and (by extrapolation) ounces. We'll work with the data there to create new fields for that information. Another is that there is a null value in the Caffeine column that we'll want to review. The Caffeine column also denotes 'Varies' instead of providing the caffeine content for several of the teas. This is likely due to the different types of tea available, ranging from a caffeine-free herbal tea to a moderately caffeinated black tea. For our simple analysis we will change these to a 0 for the herbal Passion teas, but in a future analysis we could investigate further and created new rows for the Zen Green tea and Awake Black tea options. To provide some context, when this data was aggregated there was no 'Trenta' size for iced beverages. Hot beverages will come in Short, Tall, Grande, and Venti sizes. Iced beverages will be Tall, Grande, and Venti. Ounces are generally the same, with Short = 8 oz, Tall = 12 oz, and Grande = 16 oz. Venti deviates, with a Hot Venti = 20 oz and an Iced Venti = 24 oz. Espresso drinks may come in a 'Solo' size of 1 oz, or a 'Doppio' size with 2 oz. Both Solo and Doppio will be served in a Short cup. Milk types may be changed based on the beverage, such as a Latte with multiple milk choices. Beverages may also contain no milk at all, like an espresso or brewed coffee.

# We'll define a function to identify milk types, then extract that key phrase to a new 'Milk Type' field.
	
	def extract_dairy_type(text, key_phrase):
		pattern = '|'.join(key_phrase)
		match = re.search(pattern, text, flags=re.IGNORECASE)
		if match:
			return match.group(0)
		else:
			return 'DairyFree'
	
	
	key_phrase = ['2% Milk','Nonfat Milk','Whole Milk','Soymilk']
	
	# Run dairy_extract
	
	sbux_data['Dairy Type'] = sbux_data['Beverage Prep'].apply(lambda x: extract_dairy_type(x, key_phrase))

		sizes = ['Short', 'Tall', 'Grande', 'Venti', 'Doppio', 'Solo']

# Any description that doesn't already include the size will need to be calculated. 
	# The previously created 'Rank' column can be used to assign size based on the calorie content.
	
	def determine_size(row):
		if row['MenuSize']:
			return row['MenuSize']
		else:
			if row['Sort Count'] == 1:
				return 'Grande'
			elif row['Sort Count'] != 1:
				if row['Rank'] == 4.0:
					return 'Short'
				if row['Rank'] == 3.0:
					return 'Tall'
				if row['Rank'] == 2.0:
					return 'Grande'
				if row['Rank'] == 1.0:
					return 'Venti'
				else:
					return 'ERROR'
				
	sbux_data['DrinkSize'] = sbux_data.apply(determine_size, axis=1)

# Once we know if a beverage is hot or iced it is possible to appropriately assign the ounces.
	
	def determine_ounces(row):
		if (row['Beverage Temp'] == 'Iced') and (row['DrinkSize'] == 'Venti'):
			return 24
		if row['DrinkSize'] == 'Grande':
			return 16
		if row['DrinkSize'] == 'Tall':
			return 12
		if row['DrinkSize'] == 'Short':
			return 8
		if row['DrinkSize'] == 'Doppio':
			return 2
		if row['DrinkSize'] == 'Solo':
			return 1
		if row['DrinkSize'] == 'Venti' and row['Beverage Temp'] == 'Hot':
			return 20
		else:
			return 'ERROR'
				
	sbux_data['Drink Ounces'] = sbux_data.apply(determine_ounces, axis=1)

Now we have a technically cleaned data table, ready for analysis. Technically. Where we could go wrong is by neglecting some business knowledge and assuming that all remaining records are equal. At Starbucks the default size for almost every drink is a Grande. If you load up the app and place an order if will most likely auto-select Grande as this is what most customers are ordering. Noteable exceptions are the Espresso, which is typically in a Solo or Doppio (or even Triple or Quad). The other exception is the Flat White, which is typically a Tall. This dataset does not have the Flat White, but it does have the basic Espresso shot. For an Apples-Apples comparison across beverages, and to not skew our counts based on which items come in more sizes than others, let's remove all sizes that are not Grande. Yes, we sacrificed the humble Espresso, but it would probably not survive an outliers culling at a later stage in the analysis anyway thanks to it's low price, low volume, and limited nutritional content.

Two things should be immediately noticable. One is that the total counts jumped up dramatically. This is because we are now including all size options for all drinks. The count of total items has increased because of the increased possible combinations. The other item that jumps out is that our rainbow is now out of sync. We intentionally kept the colors the same to visually identify changes. Adding in the additional size options bumped up the Signature Espresso Drinks and the Shaken Iced Beverages. Because the Smoothies only come in Grande they took a tumble. Is this visual incorrect compared to the first? That depends. If the analysis is on every permutation of the available drinks then this would get us closer to a correct count. Since we've already determined that there are gaps we elected not to pursue (beverages that could be either iced or hot, and the additional tea options with caffeine), we would be understating our counts. If the analysis is on base beverage types then it would be an overstatement of the menu. For a cleaner and more equitable understanding of nutitrional value by beverage type we'll continue to utilize the Grande only dataset.
